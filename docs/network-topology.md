## 网络拓扑与问题原理

### 网络架构

```
┌─────────────────────────────────────────────────────────┐
│                    局域网 192.168.31.0/24                 │
│                                                         │
│   你的电脑 ──────────── Unraid 宿主机 ──────── 路由器     │
│                      192.168.31.139                      │
│                          (br0)                           │
└──────────────────────────┬──────────────────────────────┘
                           │
                    ┌──────┴──────┐
                    │   docker0    │
                    │ 172.17.0.1   │
                    └──────┬──────┘
                           │
                ┌──────────┴──────────┐
                │   Docker 容器        │
                │   172.17.0.29        │
                │                      │
                │  ┌────────────────┐  │
                │  │  gost :8988    │  │
                │  │  (HTTP 代理)    │  │
                │  └────────────────┘  │
                │                      │
                │  ┌────────────────┐  │
                │  │  openconnect   │  │
                │  │  tun0 ──→ VPN  │──────→ VPN 服务器 (192.168.10.56)
                │  └────────────────┘  │
                └─────────────────────┘
```

### 问题：为什么外部 IP 无法访问？

VPN 连接前，容器的路由表：

```
Destination     Gateway         Iface
0.0.0.0         172.17.0.1      eth0    ← 默认路由走 Docker 网关
172.17.0.0/16   0.0.0.0         eth0    ← Docker 桥接网络
```

VPN 连接后，openconnect 劫持了默认路由：

```
Destination     Gateway         Iface
0.0.0.0         192.168.10.56   tun0    ← 默认路由被 VPN 劫持！
172.17.0.0/16   0.0.0.0         eth0    ← Docker 桥接网络（未变）
```

当宿主机 `192.168.31.139` 通过端口映射访问容器 `8988` 时：

```
你的电脑 telnet 192.168.31.139:8988
        │
        ▼
┌─ Unraid 宿主机 ─────────────────────────────────────────┐
│  iptables DNAT: 192.168.31.139:8988 → 172.17.0.29:8988  │
│  ✅ 请求成功转发到容器                                     │
└──────────────────────────┬──────────────────────────────┘
                           │
                           ▼
┌─ Docker 容器 ────────────────────────────────────────────┐
│  gost 收到请求，准备回复                                    │
│                                                          │
│  回包目标: 192.168.31.xxx (你的电脑)                       │
│  查路由表:                                                │
│    0.0.0.0 → 192.168.10.56 (tun0)  ← 默认路由被 VPN 劫持！│
│    172.17.0.0/16 → eth0             ← 只有这条走 eth0     │
│                                                          │
│  192.168.31.xxx 不匹配 172.17.0.0/16                      │
│  ❌ 回包走了 tun0 → VPN 隧道 → 丢失！                      │
└──────────────────────────────────────────────────────────┘
```

**根因**：OpenConnect VPN 连接后，把容器的**默认路由改成了 `tun0`**。回包目标 `192.168.31.x` 不在 Docker 网段 `172.17.0.0/16` 内，所以走了 VPN 隧道，永远到不了你的电脑。

### 解决方案：iproute2 策略路由

使用 **策略路由（policy routing）** 让容器自身发出的回包走 Docker 网关，VPN 代理流量不受影响：

```
┌─ Docker 容器（修复后）──────────────────────────────────────┐
│                                                            │
│  路由表（主表）:                                             │
│    0.0.0.0 → 192.168.10.56 (tun0)    ← VPN 流量正常走这里   │
│    172.17.0.0/16 → eth0               ← Docker 内部流量     │
│                                                            │
│  路由表 100（新增）:                                         │
│    0.0.0.0 → 172.17.0.1 (eth0)       ← 回包走 Docker 网关   │
│                                                            │
│  策略路由规则（新增）:                                        │
│    from 172.17.0.29 → 查 table 100                          │
│    含义：所有从容器 eth0 IP 发出的包，走 table 100            │
│                                                            │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ gost 收到请求，回复给 192.168.31.xxx                  │   │
│  │                                                     │   │
│  │ 回包源地址: 172.17.0.29 (容器 eth0 IP)               │   │
│  │ 命中策略规则: from 172.17.0.29 → table 100           │   │
│  │ table 100 默认路由: → 172.17.0.1 (eth0)             │   │
│  │ ✅ 回包走 eth0 → docker0 → 宿主机 iptables NAT      │   │
│  │ ✅ 宿主机把回包发给你的电脑                            │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                            │
│  同时 VPN 代理流量不受影响:                                   │
│  │ 浏览器 → gost → openconnect → tun0 → VPN 服务器 ✅      │
└────────────────────────────────────────────────────────────┘
```

### 修改文件

| 文件 | 改动 | 原因 |
|------|------|------|
| **`Dockerfile`** | `debian:buster-slim` → `debian:bookworm-slim` | Buster 已 EOL，apt 源 404 |
| **`Dockerfile`** | 新增安装 `iproute2` | 提供 `ip` 命令支持策略路由 |
| **`entrypoint.sh`** | VPN 连接前保存网关信息 | 记住 Docker 原始网关 `172.17.0.1` |
| **`entrypoint.sh`** | VPN 连接后设置 `ip route table 100` + `ip rule` | 让容器回包走 Docker 网关而非 VPN 隧道 |

### 一句话总结

> **VPN 劫持了容器的默认路由，导致回包走错路。通过策略路由，让容器自身发出的回包走 Docker 网关回去，VPN 代理流量不受影响。**
